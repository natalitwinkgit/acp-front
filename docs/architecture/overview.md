# Огляд Архітектури (Architecture Overview)

Цей документ описує високорівневу архітектуру проекту `acp-front`, який розроблений на базі **Next.js 16 (App Router)** та React 19.

## 1. Загальна концепція (Next.js App Router)

Додаток використовує новітню парадигму маршрутизації Next.js — **App Router**. Це означає, що вся маршрутизація та основні макети сторінок знаходяться в директорії `app/`.

### Особливості реалізації маршрутизації:
- **Багатомовність (i18n):** Використовується динамічний сегмент `[locale]` у корені директорії `app/`. Це дозволяє обслуговувати різні мовні версії сайту (наприклад, `/en/login` та `/uk/login`).
- **Перехоплення маршрутів (Intercepting Routes) та Паралельна маршрутизація (Parallel Routes):** В папці `app/[locale]/@modal` реалізована логіка відображення модальних вікон авторизації. Це дозволяє показувати сторінки логіну/реєстрації як модальні вікна поверх поточної сторінки, зберігаючи при цьому можливість прямого переходу на ці сторінки за окремим URL (фолбек на сторінки з `app/[locale]/(auth)`).
- **Групи маршрутів (Route Groups):** Використовуються папки з круглими дужками (наприклад, `(auth-modal)` або `(auth)`), щоб організувати файли без впливу на структуру URL.

## 2. Server Components (RSC) vs Client Components

Згідно з ідеологією Next.js App Router, архітектура додатку строго поділяється на серверні та клієнтські компоненти.

### React Server Components (RSC) - За замовчуванням
Всі компоненти в директорії `app/` та більшість компонентів `pages-layer` є серверними компонентами. 
**Відповідальність:**
- Пряма робота з бекендом (запити до API через `fetch`).
- Робота з файловою системою, змінними оточення, кешуванням.
- SEO-оптимізація (генерація метаданих через `generateMetadata`).
- Передача готового HTML та необхідних даних (пропсів) у клієнтські компоненти.

### Client Components (`"use client"`)
Використовуються тільки тоді, коли потрібна інтерактивність на стороні браузера. Зазвичай це рівень `features` або `widgets` з FSD, та інтерактивні елементи UI (кнопки з onClick, поля вводу, форми).
**Коли застосовуються:**
- Доступ до браузерних API (window, localStorage).
- Використання React хуків (`useState`, `useEffect`, `useContext`).
- Обробка подій користувача (`onClick`, `onChange`).

## 3. Патерн Feature-Sliced Design (FSD)

Попри те, що маршрутизація знаходиться в папці `app/`, **вся бізнес-логіка, UI-компоненти та інтеграції винесені в папку `src/`** згідно з методологією Feature-Sliced Design.
Це робить код незалежним від фреймворку та полегшує масштабування.

> Детальний опис розподілу по шарах FSD описаний у файлі [fsd.md](./fsd.md).

## 4. Зв'язок App Router та FSD

Щоб поєднати жорстку файлову маршрутизацію Next.js та модульність FSD, використовується наступний підхід:
- Файли `page.tsx` або `layout.tsx` в директорії `app/` є виключно "тонкими" обгортками (entry points). Вони не містять складної логіки чи розмітки.
- Кожна `page.tsx` імпортує композитний компонент сторінки (PageContent) з шару `pages-layer` (або `widgets`/`features`), де вже сконцентрована основна структура та бізнес-логіка.

**Приклад флоу:**
`app/[locale]/(auth)/login/page.tsx` → `app/[locale]/(auth)/login/LoginPageContent.tsx` → `src/widgets/BookingForm` + `src/features/auth` + `src/shared/ui/...`

## 5. Робота зі станом (State Management)

- **Локальний стан:** Використовується стандартний React стейт (`useState`, `useReducer`) всередині клієнтських компонентів для контролю UI-логіки.
- **Серверний стан:** Замість традиційних клієнтських fetch-запитів, перевага надається отриманню даних на сервері (Server Components).
- **Глобальний стан (Context):** Використовується точково (наприклад, для i18n або глобального контексту авторизації `src/features/auth/model/session.tsx`).

## 6. Локалізація (i18n)

Локалізація вбудована глибоко в архітектуру:
- Конфігурація i18n знаходиться в `src/shared/i18n/`.
- Використовуються словники (dictionaries) для різних мов: `src/shared/locales/en.json` та `uk.json`.
- Впроваджено власну обгортку для `Link` та роутингу (`src/shared/i18n/Link.tsx`), яка автоматично підставляє правильну мову в URL.
