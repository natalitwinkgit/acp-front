# Гайдлайни з розробки (Development Guidelines)

Цей документ містить набір правил та домовленостей, яких має дотримуватись кожен розробник при написанні коду для проекту `acp-front`.

## 1. Конвенції щодо іменування (Naming Conventions)

- **Компоненти (React Components):** `PascalCase`. Наприклад: `BookingForm.tsx`, `LanguageSwitcher.tsx`.
- **Файли стилів (CSS Modules):** `[ComponentName].module.css`. Наприклад: `BookingForm.module.css`.
- **Хуки (Hooks):** `camelCase` з префіксом `use`. Наприклад: `useBookingTrips.ts`.
- **Звичайні функції та утиліти:** `camelCase`. Наприклад: `formatDate.ts`, `apiClient.ts`.
- **Константи (Constants):** `UPPER_SNAKE_CASE`. Наприклад: `MAX_PASSENGERS_COUNT`.
- **Типи та Інтерфейси (Types/Interfaces):** `PascalCase`. Наприклад: `TripResponse`, `UserSession`.

## 2. Стандарти написання компонентів

1. **Функціональні компоненти:** Використовуйте тільки функціональні компоненти зі стрілочними функціями або `function` деклараціями. Класові компоненти заборонені.
2. **Типізація:** Всі компоненти повинні мати типізовані `Props`. Уникайте використання типу `any`. Використовуйте `unknown`, якщо тип справді невідомий.
3. **Експорт:** Використовуйте іменований експорт (Named exports) для кращого автодоповнення та рефакторингу. Default export дозволено лише для сторінок `page.tsx` та макетів `layout.tsx` (вимога Next.js).
4. **"use client":** Додавайте директиву `"use client"` лише на самому верху файлу і тільки тоді, коли компонент використовує стан (`useState`), ефекти (`useEffect`) або браузерні API. За замовчуванням всі компоненти є серверними.

## 3. Стилізація (Styling)

У проекті комбінується використання **CSS Modules** та **Tailwind CSS v4**.

- **CSS Modules (`*.module.css`):** Використовуються для складної, специфічної для компонента ізольованої логіки стилів, де Tailwind стає занадто громіздким, або для використання специфічних селекторів.
- **Tailwind CSS:** Використовується для швидкої розкладки (flex, grid, margins, paddings, typography). Переважно застосовується прямо в `className`.
- **Змішування:** Дозволяється поєднувати обидва підходи через пакет `clsx` або `classnames`.

```tsx
import clsx from 'clsx';
import styles from './MyComponent.module.css';

export const MyComponent = ({ isActive }) => (
  <div className={clsx('flex items-center p-4', styles.customBorder, isActive && styles.active)}>
    ...
  </div>
);
```

## 4. Конвенції комітів (Commit Conventions)

Ми використовуємо **Conventional Commits**. Для зручності налаштований пакет `commitizen`.
**Замість стандартного `git commit` обов'язково використовуйте:**
```bash
pnpm commit
```

Структура повідомлення коміту:
`<type>(<scope>): <subject>`

**Допустимі типи (types):**
- `feat`: Новий функціонал.
- `fix`: Виправлення багів.
- `docs`: Зміни виключно в документації (як цей файл!).
- `style`: Зміни, що не впливають на логіку (пробіли, форматування, відсутні крапки з комою).
- `refactor`: Зміни коду, які не виправляють помилки і не додають функціоналу.
- `perf`: Зміна коду, що покращує продуктивність.
- `test`: Додавання або виправлення тестів.
- `chore`: Зміни в процесі збірки, допоміжних інструментах тощо.

## 5. Робота з гілками та Pull Requests (PR)

1. **Гілкування:** Створюйте гілки від `main` (або `develop`, залежно від поточного флоу).
2. **Іменування гілок:** `type/task-number-short-description`.
   - Приклад: `feat/ACP-123-booking-form` або `fix/header-mobile-menu`.
3. **Pull Request:** 
   - PR повинен мати чіткий опис того, що було зроблено.
   - Обов'язково перевірте код лінтером (`pnpm lint`) та переконайтесь, що проект збирається (`pnpm build`) перед створенням PR.
   - Залишайте код у кращому стані, ніж ви його знайшли (Boy Scout Rule).
